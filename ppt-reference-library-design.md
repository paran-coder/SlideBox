# 슬라이드박스 — PPT 레퍼런스 라이브러리 설계 문서

## 개요

PPT 디자이너가 모아둔 레퍼런스 파일(PDF, PPTX)을 가져오고, 슬라이드 낱장 단위로 훑어보며, AI가 달아준 태그로 빠르게 다시 찾을 수 있는 웹 앱입니다.

**해결하는 문제.** 레퍼런스를 폴더에 모아두기만 하고, 정작 필요할 때 찾는 데 시간이 오래 걸립니다. 근본 원인은 정리(태깅)를 안 하게 되는 것과, 파일을 열어보기 전까지 내용이 안 보이는 것입니다.

**핵심 결정 사항.**
- 대상: 일단 제작자 본인용, 검증되면 제품화.
- 검색 축: 스타일 / 레이아웃 / 주제 3축 (찾을 때 세 가지가 섞여 떠오르기 때문).
- 검색 단위: 파일 단위 + 슬라이드 낱장 단위 모두.
- 태깅: AI가 초안을 달고, 사용자가 수정·보완하는 하이브리드.
- **완전한 로컬 저장 구조.** 사용자가 자기 컴퓨터의 폴더 하나를 "라이브러리 폴더"로 지정하면, 그 폴더가 원본 파일·썸네일·메타데이터를 모두 담는 저장소가 됩니다. 서버, 데이터베이스, 클라우드 계정이 전혀 필요 없습니다.
- **AI 태깅만 외부와 통신합니다.** 사용자의 Anthropic API 키(BYOK)로 브라우저에서 Anthropic에 직접 호출합니다. 이 통신을 제외하면 앱은 사용자의 디스크 밖으로 아무것도 보내지 않습니다.
- **사용자가 준비할 것은 정확히 두 가지입니다.** 라이브러리 폴더 선택(클릭 한 번)과 Anthropic API 키 입력.
- PDF-PPTX 쌍은 확장자를 뺀 파일명(file_key)으로 자동 매칭.
- 가져오기는 단일 추가가 기본, 일괄 추가 모드 지원.
- PPTX→PDF 변환 기능은 MVP에서 제외(확장 후보). 사용자는 PowerPoint의 PDF 내보내기를 사용.
- 초기 규모: 수십 개 파일.

**중대한 제약 사항 — 반드시 인지하고 결정한 사항입니다.** 이 구조는 브라우저의 File System Access API(`showDirectoryPicker`)에 기반합니다. 이 API는 Chrome 86+, Edge 86+, Opera 72+ 등 **Chromium 계열 데스크톱 브라우저에서만 동작합니다.** Firefox는 이 방식의 로컬 디스크 피커를 지원하지 않기로 자체 정책으로 결정했고, Safari는 macOS·iOS 어디에서도 지원하지 않으며, 모바일 브라우저는 플랫폼을 가리지 않고 전부 지원하지 않습니다. 즉 **이 앱은 PC의 Chrome 또는 Edge에서만 쓸 수 있습니다.** 지금 단계(제작자 본인용, PC 사용)에는 문제가 없다고 판단해 이 제약을 받아들이기로 결정했으나, 나중에 제품화해서 Safari·모바일 사용자를 받아야 한다면 이 구조를 다시 검토해야 합니다.

**되돌린 결정들 (경로 변경 히스토리).**
- Supabase(운영자 소유 중앙 DB) → Supabase(각 사용자 소유 BYO 프로젝트) → **로컬 폴더**로 최종 정착했습니다. 사용자 데이터를 운영자가 절대 접근할 수 없어야 한다는 요구를 가장 확실하게, 그리고 온보딩 부담 없이 만족시키는 방법이었기 때문입니다.
- 이 과정에서 로그인, user_id, 연결 마법사, RLS 정책이 모두 불필요해져 제거했습니다.
- 메타데이터 내보내기/가져오기 기능도 불필요해져 제거했습니다(아래 "내보내기·가져오기가 필요 없는 이유" 참조).

## 접근 방식

**라이브러리 폴더가 곧 저장소입니다.**

사용자가 설정 화면에서 폴더 하나를 지정하면, 앱은 그 폴더 안에 다음 구조를 관리합니다.

```
<지정한 폴더>/
  library.json              # 전체 메타데이터(제목, 메모, 태그, 슬라이드 목록)
  originals/
    {file_key}.pdf
    {file_key}.pptx          # 있는 경우만
  thumbs/
    {file_key}/
      p1.jpg
      p2.jpg
      ...
```

- 파일명을 `file_key` 그대로 쓰기 때문에, 사용자가 탐색기·Finder로 이 폴더를 직접 열어봐도 무엇이 들어있는지 바로 알아볼 수 있습니다.
- `library.json` 하나가 데이터베이스 역할을 전부 합니다. 수십 개 파일, 수백 개 슬라이드 규모에서는 이 파일 전체를 읽고 쓰는 방식이 충분히 빠르고 훨씬 단순합니다.
- 이 폴더를 백업하거나 다른 컴퓨터로 옮기는 일은 사용자가 이미 익숙한 방식(복사, 클라우드 동기화)으로 그대로 됩니다. **폴더를 Dropbox나 OneDrive처럼 동기화되는 위치에 지정하면, 앱이 아무것도 하지 않아도 여러 기기 간 동기화가 자연히 이루어집니다.** 이건 설계상 의도한 기능은 아니지만 이 구조의 자연스러운 부수 효과입니다.

**PDF는 보는 용도, PPTX는 보관용.**

- PDF를 가져오면 브라우저에서 슬라이드 낱장 썸네일로 변환해 `thumbs/`에 저장합니다. AI 태깅도 이 이미지 기준입니다.
- PPTX는 변환 없이 `originals/`에 원본 그대로 저장하고, 필요할 때 열어봅니다.
- 확장자를 뺀 파일명이 같으면(예: `강남프로젝트_제안서.pdf` + `강남프로젝트_제안서.pptx`) 한 쌍으로 인식해 하나의 레퍼런스로 묶습니다.
- PPTX만 나중에 가져와도 file_key가 일치하면 기존 레퍼런스에 자동 연결됩니다.
- 같은 file_key를 다시 가져오면 "덮어쓰기 / 건너뛰기"를 확인합니다.
- 트레이드오프: PPT의 애니메이션·인터랙션은 PDF에서 안 보입니다. 정적 비주얼 참고가 주 용도이므로 초기에는 허용합니다.

**PDF는 "열기" 버튼, PPTX는 클릭 복사입니다.** PDF는 브라우저가 바로 렌더링할 수 있어서 "PDF 열기" 버튼을 누르면 파일을 읽어 새 탭에서 보여줍니다. PPTX는 브라우저가 형식을 렌더링할 수 없어서, 버튼을 눌러도 다운로드 사본이 생길 뿐입니다. 원본이 이미 라이브러리 폴더 안에 있는데 다운로드 폴더에 사본을 또 만드는 건 의미가 없다고 판단해, PPTX는 버튼 대신 **폴더 내 경로 텍스트**(예: `originals/강남프로젝트_제안서.pptx`)를 보여주고, 클릭하면 그 경로가 클립보드로 복사되어 사용자가 자기 파일 탐색기·Finder의 주소창에 붙여넣어 바로 찾아갈 수 있습니다.

**수정 사항은 즉시 저장됩니다.** 제목·메모·태그 편집에는 별도의 저장 버튼이 없습니다. 값이 바뀌는 즉시 `library.json`에 기록되고, 저장이 끝나면 "저장됨" 같은 짧은 안내가 화면에 잠깐 표시되어 사용자가 저장 여부를 눈으로 확인할 수 있습니다.

**AI 태깅만 브라우저에서 외부로 나갑니다 (BYOK).**

- 사용자가 설정 화면에 자기 Anthropic API 키를 입력하면 localStorage에 저장되고, 이후 슬라이드 이미지를 Anthropic에 직접 전송해 태그 초안을 받습니다.
- 브라우저 직접 호출이라 서버가 필요 없고, 서버리스 타임아웃 제약도 없습니다.
- 이 키는 localStorage에만 있고 운영자 서버로 전송되는 경로가 없습니다. "비밀번호처럼 다루라"는 경고를 설정 화면에 명시합니다.

## 화면 구성 (4개 화면)

1. **라이브러리(홈).** 슬라이드 썸네일이 Pinterest식 그리드로 펼쳐지는 메인 화면. 상단에 검색창 + 태그 필터. "파일 보기 / 슬라이드 보기" 전환 토글로 두 단위를 오갑니다. 라이브러리 폴더가 연결되지 않았으면 설정 화면으로 안내합니다.
2. **레퍼런스 상세.** 파일 하나를 열면 전체 슬라이드 목록, AI가 단 태그(수정 가능), 메모, "PDF 열기" 버튼, PPTX가 있으면 폴더 내 경로가 표시됩니다. 제목·메모·태그를 수정하면 별도의 저장 버튼 없이 즉시 저장되며, 저장이 끝나면 화면 한쪽에 "저장됨" 같은 짧은 안내가 잠깐 표시됩니다. PPTX 경로는 텍스트를 클릭하면 클립보드로 복사되고, 복사되었다는 안내가 함께 표시됩니다.
3. **가져오기.** 단일 모드(기본)와 일괄 모드. PDF(필수) + PPTX(선택)를 선택하면 file_key로 자동 짝지어지고, 변환 → 폴더에 복사·저장 → AI 태그 초안 생성 → 확인 후 저장. 일괄 모드에서는 파일별 진행 상태를 목록으로 표시하며, "AI 태깅 지금 실행 / 나중에" 선택으로 크레딧 사용을 제어합니다. API 키가 없으면 태깅 옵션은 비활성화되고 설정으로 안내합니다.
4. **설정.** (a) 라이브러리 폴더 선택·재연결(브라우저 재시작 후 접근 권한을 다시 확인해야 할 수 있습니다) (b) Anthropic API 키 입력(마스킹, 발급 가이드, 월 지출 한도 권장). 모든 키는 "비밀번호처럼 다루고 공유하지 마세요" 경고를 명시합니다.

## 데이터 구조 (library.json)

```json
{
  "version": 1,
  "tags": [
    { "id": "style-minimal", "name": "미니멀", "kind": "style", "is_preset": true }
  ],
  "refs": [
    {
      "id": "...",
      "file_key": "강남프로젝트_제안서",
      "title": "강남 프로젝트 제안서",
      "memo": "표지 그라데이션 참고용",
      "has_pptx": true,
      "created_at": "2026-07-05T...",
      "tag_ids": ["style-minimal"],
      "slides": [
        { "page_no": 1, "thumb": "thumbs/강남프로젝트_제안서/p1.jpg", "tag_ids": ["layout-cover"] }
      ]
    }
  ]
}
```

- 태그 사전(스타일 15개, 레이아웃 12개, 주제 10개)은 폴더를 처음 연결할 때 `library.json`에 프리셋으로 미리 채워둡니다.
- 태그는 파일(ref)과 슬라이드 양쪽에 `tag_ids`로 붙습니다.
- 이 파일은 앱이 전체를 읽어 상태로 올리고, 변경이 있을 때마다 전체를 다시 씁니다. 규모가 수십~수백 슬라이드 수준이라 이 방식이 가장 단순하고 충분합니다.
- 동시 저장 충돌(여러 탭에서 동시 저장 등)은 마지막에 저장한 내용이 남는 방식으로 처리합니다. 개인용 단일 탭 사용을 전제로 하므로 별도 잠금 장치는 두지 않습니다.

## AI 태깅 흐름

- 가져오기 시 각 슬라이드 이미지를 브라우저에서 사용자의 API 키로 Claude API에 보내 태그 초안을 받습니다. 일괄 가져오기에서는 "나중에"를 선택해 건너뛸 수 있고, 상세 화면에서 언제든 실행할 수 있습니다.
- **태그 사전 규칙.** AI는 미리 정의된 태그 사전 안에서만 고릅니다. 자유 생성을 허용하면 유사 태그가 난립해 필터가 망가지기 때문입니다. 사전에 없는 새 태그는 사용자만 추가할 수 있습니다.
- AI 결과는 "초안" 상태로 저장되고, 상세 화면에서 클릭 한 번으로 빼거나 추가할 수 있습니다.

## 내보내기·가져오기가 필요 없는 이유

이전 설계에서는 원격 DB가 초기화되거나 이전될 때 AI 크레딧을 다시 쓰지 않도록 메타데이터 내보내기/가져오기 기능을 두었습니다. **로컬 폴더 구조에서는 이 문제 자체가 발생하지 않습니다.** `library.json`이 태그·메모를 원본 파일과 같은 폴더에 항상 함께 보관하기 때문에, 폴더를 통째로 지우지 않는 한 데이터가 초기화될 일이 없습니다. 다른 컴퓨터로 옮기고 싶으면 폴더를 그대로 복사하면 되고, 이는 사용자가 이미 알고 있는 일반적인 파일 복사 동작입니다. 이 기능은 MVP에서 만들지 않습니다.

## 검색·필터 동작

- 검색창 키워드는 제목, 메모, 태그 이름을 한 번에 검색합니다.
- 태그 필터는 스타일 / 레이아웃 / 주제 3축이 나란히 놓이고, 복수 선택 시 교집합으로 좁혀집니다.
- 파일 보기 모드는 파일 태그 기준, 슬라이드 보기 모드는 슬라이드 태그 기준으로 필터가 작동합니다.
- `library.json` 전체를 한 번에 불러와 클라이언트에서 검색·필터링합니다. 규모가 커지면 그때 다른 방식을 고려합니다.

## 기술 스택

- **Next.js.** 화면 전용. 서버 API 라우트는 없습니다.
- **File System Access API.** 라이브러리 폴더 접근, 읽기·쓰기. Chromium 계열 데스크톱 전용.
- **pdf.js.** 브라우저에서 PDF → 슬라이드 이미지 변환.
- **Claude API (BYOK).** 브라우저에서 사용자 키로 직접 호출.
- **Vercel.** 배포. 환경변수 없음.

운영자가 준비할 것은 Vercel 배포뿐입니다. Supabase, 데이터베이스, 서버 비밀값이 전혀 없습니다.

## 보안 원칙

- Anthropic API 키는 localStorage에만 저장되며, 운영자 서버로 전송되는 코드 경로가 존재하지 않습니다.
- 라이브러리 폴더에 대한 접근 권한은 브라우저가 관리하며, 사용자가 명시적으로 선택한 폴더 외에는 앱이 디스크에 접근할 수 없습니다.
- 앱이 통신하는 외부 대상은 오직 `api.anthropic.com`뿐입니다.

## 오류 처리

- 지원하지 않는 브라우저(Firefox, Safari, 모바일)로 접속하면 진입 화면에서 즉시 "Chrome 또는 Edge 데스크톱에서 사용해 주세요"를 안내하고 나머지 기능을 막습니다.
- 라이브러리 폴더 접근 권한이 만료되었거나 거부되면(예: 브라우저 재시작 후) "폴더 접근을 다시 허용해 주세요" 버튼을 표시합니다.
- PDF 변환 실패(손상된 파일 등) 시 해당 파일만 실패로 표시하고 나머지 가져오기는 진행합니다.
- AI 태깅 실패 시 태그 없이 저장하고, 상세 화면에서 "AI 태깅 다시 시도" 버튼으로 재시도합니다. API 키가 유효하지 않으면(401) 설정 화면으로 안내합니다.
- 가져오기 용량 제한을 두고(예: 파일당 50MB), 초과 시 가져오기 전에 안내합니다.
- 같은 file_key 재가져오기 시 덮어쓰기/건너뛰기 확인. 덮어쓰기는 기존 슬라이드·태그를 교체합니다.

## 검증 기준

MVP가 성공적이라고 볼 수 있는 조건입니다.

1. Chrome 또는 Edge에서 폴더 선택 한 번으로 라이브러리 연결이 완료됩니다.
2. PDF를 가져오면 30초 내에 슬라이드 그리드에 나타납니다.
3. AI 태그 초안이 사전 안의 태그로만 달리고, 클릭으로 수정됩니다.
4. 태그 3축 복수 선택 필터가 교집합으로 정확히 작동합니다.
5. "PDF 열기" 버튼으로 PDF 원본이 새 탭에서 열립니다. PPTX가 있으면 경로 텍스트를 클릭했을 때 클립보드로 복사되고 안내가 표시됩니다.
6. 제목·메모·태그를 수정하면 별도 버튼 없이 즉시 저장되고, "저장됨" 안내가 표시됩니다.
7. 같은 file_key의 PDF와 PPTX가 자동으로 한 레퍼런스로 묶입니다.
8. 일괄 가져오기에서 파일 10개를 한 번에 올려도 개별 진행 상태가 표시되고 실패가 전체를 막지 않습니다.
9. **브라우저 네트워크 탭에서 통신 대상이 api.anthropic.com뿐임을 확인합니다.**
10. 브라우저를 재시작한 뒤 다시 접속해도(권한 재확인 클릭 한 번 후) 기존 라이브러리가 그대로 보입니다.
11. Firefox나 Safari로 접속하면 즉시 안내 문구가 표시되고 앱이 오작동하지 않습니다.
12. 제작자 본인이 실제 레퍼런스 수십 개를 넣고 "찾는 시간이 줄었다"고 느낍니다.
